# 法的リスク分析

契約書・OSSライセンス・プライバシーポリシーの法的リスクを体系的に分析する。

**⚠️ 免責事項: この分析は法的助言ではありません。正式な法的判断には弁護士への相談を推奨します。**

## 動的コンテキスト

```
リポジトリルート: !`git rev-parse --show-toplevel 2>/dev/null || echo "(リポジトリ外)"`
パッケージマネージャ: !`ls package.json Cargo.toml go.mod composer.json Gemfile requirements.txt pyproject.toml 2>/dev/null || echo "(検出なし)"`
```

---

## Phase 0: 入力検証・分類

### 0-1. 引数の確認
- `$ARGUMENTS` が空の場合 → 使い方を案内して終了:
  ```
  使い方:
    /legal-check <ファイルパス>   — 契約書・法的文書の条項リスク分析
    /legal-check license         — プロジェクト依存関係のOSSライセンス分析
    /legal-check privacy         — プロジェクトの個人情報処理パターン分析
  ```

### 0-2. 入力の分類
引数を以下のいずれかに自動分類する:

| 引数パターン | 分類 | 次のPhase |
|-------------|------|----------|
| `license` / `ライセンス` | OSSライセンス分析 | Phase 1B |
| `privacy` / `プライバシー` | プライバシー分析 | Phase 1C |
| ファイルパス（`.pdf`, `.docx`, `.txt`, `.md`等） | 契約書分析 | Phase 1A |
| それ以外 | ファイルパスとして試行 → 失敗なら案内 | — |

### 0-3. ファイル分析の場合の前処理
- 指定ファイルの存在確認（Read で読み取り）
- PDF の場合は Read ツールの pages パラメータを使用（大きなPDFは分割読み取り）
- 文書の種類を冒頭部分から判定（契約書/利用規約/プライバシーポリシー/ライセンス文書）

## Phase 1A: 契約書分析（日本法中心）

指定された文書を以下の観点でチェックする。

### 1A-1. 下請法チェック
- **3条書面の要件**: 給付内容、下請代金額、支払期日、支払方法が明記されているか
- **60日ルール**: 支払期日が受領日から60日以内に設定されているか
- **減額・返品の制限**: 不当な減額・返品条項がないか
- **買いたたきの禁止**: 通常支払われる対価と比べて著しく低い下請代金でないか

### 1A-2. 消費者契約法チェック（B2C契約の場合）
- **不当条項の検出**:
  - 事業者の損害賠償責任を全部免除する条項
  - 消費者の解除権を放棄させる条項
  - 高額な違約金・キャンセル料条項
  - 消費者の利益を一方的に害する条項
- **特定商取引法**: 表示義務事項の漏れ、クーリングオフ条項の有無

### 1A-3. 民法（契約不適合責任・定型約款）
- **契約不適合責任**: 追完請求・代金減額・損害賠償・解除の権利が不当に制限されていないか
- **定型約款**: 不当に相手方の利益を害する条項がないか（民法548条の2第2項）

### 1A-4. 一般条項チェック
- **解約条件**: 中途解約の可否、解約予告期間、解約時の精算方法
- **損害賠償**: 上限設定、間接損害の取り扱い、免責事由
- **知的財産権**: 成果物の帰属、ライセンスの範囲、第三者IPの保証
- **競業避止義務**: 期間・地域・業種の合理性
- **秘密保持**: 定義の明確性、期間、例外事由
- **反社会的勢力排除**: 条項の有無と内容の適切性
- **紛争解決**: 管轄裁判所、仲裁条項
- **契約期間と自動更新**: 自動更新条件、更新拒絶の方法と期限

## Phase 1B: OSSライセンス分析

### 1B-1. 依存関係のライセンス収集
- `~/.config/claude/skills/legal-check/scripts/scan-licenses.sh` を実行してライセンス一覧を取得
- スクリプトが取得できなかった依存関係がある場合は、手動で補完調査

### 1B-2. ライセンス互換性チェック

**コピーレフトリスク（HIGH/CRITICAL）:**
- GPL-2.0 / GPL-3.0: プロジェクト全体にコピーレフト効果あり（静的リンクの場合）
- AGPL-3.0: ネットワーク経由の利用でもコピーレフト効果あり
- LGPL: 動的リンクなら通常OK、静的リンクは注意

**注意が必要（MEDIUM）:**
- MPL-2.0: ファイル単位のコピーレフト
- Apache-2.0: 特許条項あり（特許訴訟時にライセンス終了）
- EPL: GPL非互換

**低リスク（LOW）:**
- MIT / BSD-2-Clause / BSD-3-Clause / ISC / Unlicense / CC0

**要調査（CRITICAL）:**
- UNLICENSED / ライセンス不明 / カスタムライセンス

### 1B-3. 帰属表示義務の確認
- NOTICE ファイルの必要性（Apache-2.0等）
- 著作権表示の要件
- ライセンス文の同梱義務

### 1B-4. プロジェクトライセンスとの互換性
- プロジェクト自体の LICENSE ファイルを確認
- 依存関係のライセンスとの互換性を判定

## Phase 1C: プライバシー分析

### 1C-1. コード内の個人情報処理パターン検出
Grep で以下のパターンを検索:
- **メールアドレス**: `email`, `mail_address`, `メール`
- **氏名**: `name`, `first_name`, `last_name`, `姓`, `名`
- **電話番号**: `phone`, `tel`, `電話`
- **住所**: `address`, `zip`, `postal`, `住所`, `郵便番号`
- **生年月日**: `birthday`, `birth_date`, `生年月日`
- **IPアドレス/デバイス情報**: `ip_address`, `user_agent`, `device_id`
- **位置情報**: `latitude`, `longitude`, `geolocation`, `位置`
- **Cookie/トラッキング**: `cookie`, `tracking`, `analytics`, `gtag`, `ga(`
- **認証情報**: `password`, `token`, `session`, `credential`
- **決済情報**: `credit_card`, `payment`, `billing`, `card_number`

### 1C-2. 個人情報保護法（2022年改正）準拠チェック
- **利用目的の特定と通知**: 利用目的がコード上で明確か
- **第三者提供**: 外部API・SDK への個人情報送信の有無
- **越境移転**: 海外サーバー・外部サービスへのデータ転送
- **要配慮個人情報**: 健康情報、信条、社会的身分等の処理有無
- **個人関連情報**: Cookie等の第三者提供時の確認義務
- **仮名加工情報/匿名加工情報**: データの加工処理の有無

### 1C-3. プライバシーポリシーとの整合性
- プロジェクト内のプライバシーポリシー文書を検索（`privacy`, `policy`, `プライバシー`）
- コードで実際に処理している個人情報とポリシー記載内容の突き合わせ
- 記載漏れ・実装との乖離を指摘

## Phase 2: レポート生成

以下の形式でリスク分析レポートを出力する:

```markdown
## 法的リスク分析レポート

### リスクサマリー

| リスクレベル | 件数 |
|------------|------|
| 🔴 高     | X件  |
| 🟡 中     | X件  |
| 🟢 低     | X件  |

### 詳細

#### 🔴 高リスク項目

##### [項目番号] {リスク名}
- **該当箇所**: {ファイル名:行番号 または 条項番号}
- **問題**: {問題の説明}
- **関連法令**: {該当する法律・条文}
- **リスク影響**: {放置した場合の影響}
- **推奨対応**: {具体的な対応方法}

#### 🟡 中リスク項目
（同形式で記載）

#### 🟢 低リスク項目
（同形式で記載）

---
⚠️ **免責事項**: この分析はAIによる参考情報であり、法的助言ではありません。
法的判断や契約締結の前に、必ず弁護士等の専門家にご相談ください。
特に高リスク項目については、早急な専門家への相談を推奨します。
```

## Phase 3: 修正提案（承認ゲート）

レポート出力後、AskUserQuestion で次のアクションを提示する:

```
分析が完了しました。次のアクションを選択してください:

1. 高リスク項目の修正案を詳細に提示
2. 特定の項目を深掘り分析
3. チェックリスト形式で対応表を生成
4. 比較分析（Phase 4）に進む（2つの文書バージョンを比較）
5. 分析終了
```

- ユーザーが選択するまで待機する。
- 修正案は「コード/文書の具体的な修正テキスト」ではなく、「どのような修正が必要か」の指針を示す。

---

## Phase 4: 比較分析（オプション・承認ゲート）

2つのバージョンの文書やライセンス状態を比較し、リスクの変化を分析する。

### 4-1. 入力の確認
AskUserQuestion で比較対象を確認:
- 契約書: 旧バージョンと新バージョンのファイルパス
- ライセンス: 2つのブランチ/コミット間の依存関係差分
- プライバシー: 2バージョン間のコード差分

### 4-2. 差分リスクレポート
```markdown
### リスク変化サマリー

| 変化 | 件数 |
|------|------|
| 🆕 新規リスク | X件 |
| ✅ 解消されたリスク | X件 |
| ⚠️ リスクレベル変化 | X件 |
| ➡️ 変化なし | X件 |
```

## 制約事項（厳守）

- **読み取り専用**: このSkillはファイルの分析・調査に特化する。ファイルの編集は行わない。
- **Git操作禁止**: commit, push, reset 等の状態変更コマンドは実行しない。
- **免責事項必須**: すべてのレポート出力に免責事項を含める。
- **推測の明示**: 法的判断で確証がない場合は「可能性がある」「要確認」等と明示する。
- **機密情報配慮**: 分析対象文書の機密情報を不必要に出力しない。
- **過度な断定の回避**: 「違法である」等の断定的表現は避け、「リスクがある」「抵触する可能性がある」と表現する。
