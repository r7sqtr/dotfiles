# コードレビュー

指定されたコードまたは差分を段階的に分析し、実用的な指摘を行う。

## 動的コンテキスト

```
リポジトリルート: !`git rev-parse --show-toplevel 2>/dev/null || echo "(リポジトリ外)"`
現在のブランチ: !`git branch --show-current 2>/dev/null || echo "(detached)"`
未コミット変更: !`git diff --stat HEAD 2>/dev/null | tail -1 || echo "(変更なし)"`
```

---

## Phase 0: 入力の検証とモード判定

### 0-1. 引数の解析

`$ARGUMENTS` を以下のルールで解析する:

| 引数パターン | モード | 対象 |
|-------------|--------|------|
| `src/auth.ts` 等のパス | ファイルレビュー | 指定ファイル/ディレクトリ |
| `--diff` | diff レビュー | 未コミット変更全体（`git diff HEAD`） |
| `--diff HEAD~3` 等 | diff レビュー | 指定範囲の差分（`git diff HEAD~3`） |
| `--diff --staged` | diff レビュー | ステージ済みのみ（`git diff --staged`） |
| (空) | 自動判定 | 未コミット変更があれば diff モード、なければエラー |

### 0-2. 対象の取得

- **ファイルモード**: Read で対象ファイルを読み取る。ディレクトリの場合は Glob でファイル一覧を取得し、主要ファイルを読む。
- **diff モード**: `git diff` で差分を取得。変更ファイルが多い場合は、変更量の大きいファイルから優先的にレビューする。
- **引数が空で変更もない場合** → 「レビュー対象を指定してください」と使い方を案内して終了:
  ```
  使い方:
    /review src/auth.ts       — 指定ファイルをレビュー
    /review src/              — ディレクトリ内の主要ファイルをレビュー
    /review --diff            — 未コミット変更をレビュー
    /review --diff HEAD~3     — 指定範囲の差分をレビュー
    /review --diff --staged   — ステージ済みの変更をレビュー
  ```

### 0-3. 言語・フレームワークの特定

ファイル拡張子・import文・設定ファイルから言語とフレームワークを特定する。これにより以降の分析観点を調整する。

---

## Phase 1: 全体構造の把握

コードを読み込む前に、まず全体像を理解する:

- **目的**: このコード/変更は何を達成しようとしているか
- **スコープ**: 影響範囲はどこまでか（ファイルモード: 関連モジュール、diff モード: 変更ファイル一覧）
- **データフロー**: 入力 → 処理 → 出力の流れ

---

## Phase 2: 段階的分析

以下の6観点から、**対象コードに関連するもののみ**を選択して分析する。全観点を毎回網羅する必要はない。

### 観点1: バグ・ロジック正確性
- 境界値・エッジケースの考慮漏れ
- null/undefined/nil の未チェック
- 型の不整合・暗黙の型変換
- 条件分岐のロジック誤り（off-by-one、AND/OR の取り違え等）
- 並行処理のレースコンディション
- 状態管理の不整合

### 観点2: エラーハンドリング
- エラーの握りつぶし（空の catch、_ で無視等）
- 不十分なエラー伝搬（原因情報の消失）
- 外部API・ネットワーク通信のタイムアウト・リトライ未設定
- ユーザーへのエラーメッセージの適切性

### 観点3: セキュリティ
- 入力値のバリデーション不足（SQLインジェクション、XSS、コマンドインジェクション等）
- 機密情報のハードコード・ログ出力
- 認証・認可の不備
- 安全でない暗号化・ハッシュの使用

### 観点4: パフォーマンス
- N+1クエリ問題
- 不要な再計算・メモリアロケーション
- ループ内の重い処理（I/O、API呼び出し等）
- キャッシュの欠如または不適切な無効化

### 観点5: 設計・保守性
- 責務の分離の不備（単一責任の原則違反）
- コードの重複（DRY原則違反）
- 過度な結合・不適切な依存関係
- マジックナンバー・不明瞭な命名

### 観点6: テスタビリティ
- テストが困難な構造（グローバル状態への依存、副作用の混在等）
- モック/スタブが困難な外部依存の直接参照
- テストされていない重要な分岐パス

---

## Phase 3: レポート生成

### 重要度の分類基準

各指摘に以下のいずれかの重要度を付与する:

- **CRITICAL**: 本番でバグ・障害・セキュリティ問題を引き起こす可能性が高い
- **WARNING**: 問題の可能性あり、または保守性を著しく損なう
- **INFO**: 改善で品質向上するが、動作に支障なし

### 出力フォーマット

```markdown
## レビュー結果

### サマリー
- **対象**: {ファイル名/diff範囲}
- **言語/FW**: {検出した言語・フレームワーク}

| 重要度 | 件数 |
|--------|------|
| CRITICAL | X件 |
| WARNING  | X件 |
| INFO     | X件 |

{全体の所感 — 1-2文}

### 指摘事項

#### [CRITICAL] ファイル名:行番号 - タイトル
**問題**: {何が問題か}
**影響**: {放置した場合の影響}
**推奨**: {具体的な改善方法}

#### [WARNING] ファイル名:行番号 - タイトル
**問題**: {何が問題か}
**影響**: {放置した場合の影響}
**推奨**: {具体的な改善方法}

#### [INFO] ファイル名:行番号 - タイトル
**問題**: {何が問題か}
**推奨**: {具体的な改善方法}

### 良い実践
- {本当に良い点がある場合のみ、1-3個}
```

指摘がない場合は「レビューの結果、特に指摘事項はありません。」と明確に伝える。

---

## 品質制御ルール

- **深さ優先**: 1ファイルあたり最大5件。浅く多くではなく、深く少なく指摘する。
- **些細な指摘は省略**: スタイルの好み（セミコロン有無、引用符の種類等）は指摘しない。リンター/フォーマッターの領域。
- **推測の明示**: 確証がない推測は「可能性がある」と明示する。コンテキストが不足している場合はその旨を伝える。
- **建設的な指摘**: 批判ではなく改善提案として記述する。問題だけでなく解決策を提示する。
- **CRITICAL → WARNING → INFO 順**: 重要度の高いものから記載し、読み手が優先度を即座に判断できるようにする。
- **コードコンテキストの活用**: diff モードでは変更箇所だけでなく、必要に応じて周辺コードも Read で読み、文脈を踏まえた指摘を行う。

---

## 制約事項（厳守）

- **ファイルの編集禁止**: 指摘のみを行い、コードを書き換えない。
- **Git操作禁止**: commit, push, reset 等の状態変更コマンドは実行しない。
- **推測の明示**: 確証がない推測は必ず「可能性がある」等と明示する。
- **プライバシー配慮**: コード中の機密情報（APIキー、個人情報等）を不必要に出力しない。

## 調査の進め方

- ファイルモードでは、まず対象ファイルを通読してから分析を開始する。
- diff モードでは、差分の全体像を把握してから個別の変更を分析する。
- Agent（Explore）は、関連ファイルの広範な調査が必要な場合のみ使用する。
- 大規模なレビュー対象の場合は、変更量・複雑度の高いファイルを優先する。
