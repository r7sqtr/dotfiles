# パフォーマンス監査

コードベースのパフォーマンスボトルネックを体系的に検出し、改善提案を行う読み取り専用の分析 Skill。

## 動的コンテキスト

引数が指定された場合はその範囲を対象とする。未指定の場合はプロジェクト全体を対象とする。

---

## 処理フロー

### Phase 1: プロジェクト構成の把握

1. プロジェクトの技術スタックを特定する（言語、フレームワーク、ORM、DB）
2. 引数で指定された対象範囲を確認する
3. Agent(Explore) を使い、対象コードパスの全体像を把握する

### Phase 2: SQLクエリ分析

以下の観点でSQLクエリを検出・分析する:

1. **N+1 クエリ検出**
   - ループ内でのDB呼び出しパターンを Grep で検索
   - ORM の lazy loading が暗黙的に発動する箇所を特定
   - 検索キーワード例: `->get()`, `->find()`, `::find(`, `foreach.*->`, `\.objects\.get`, `SELECT.*WHERE` がループ内にあるか

2. **フルテーブルスキャン候補**
   - WHERE句なしの大量データ取得
   - インデックスが効かないLIKE前方一致以外のパターン (`LIKE '%...%'`)
   - 関数適用による暗黙的なインデックス無効化 (`WHERE YEAR(created_at)`)
   - 大量の `OR` 条件

3. **重いクエリパターン**
   - サブクエリのネスト
   - 不要な `SELECT *`
   - JOINの多重結合（4テーブル以上）
   - GROUP BY + HAVING の大量データ集計
   - ORDER BY でインデックス外カラム使用

### Phase 3: インデックス構造の確認

DB接続が可能な場合（ユーザーに確認の上）:
- `SHOW INDEX FROM {table}` または同等コマンドを実行
- マイグレーションファイルからインデックス定義を読み取る
- 主要クエリの WHERE/JOIN/ORDER BY カラムとインデックスの対応を確認

DB接続が不可能な場合:
- マイグレーションファイル/スキーマ定義ファイルからインデックスを読み取る
- コード上のクエリと突き合わせて不足インデックスを推定

### Phase 4: コードレベルのボトルネック検出

1. **メモリ効率**
   - 大量データの一括ロード（ページネーション/チャンク処理なし）
   - 不要なデータの保持（使わないカラムの取得）

2. **計算効率**
   - ループ内の重複計算
   - キャッシュ可能だがキャッシュされていない処理
   - 同期的な外部API呼び出しのブロッキング

3. **I/O効率**
   - ファイル処理のバッファリング不足
   - 不要なログ出力（ループ内の大量ログ）

### Phase 5: レポート生成

以下の形式で結果を提示する:

```markdown

| # | 種類 | 場所 | 問題 | 推定影響 | 改善案 |
```

---

## 制約事項

- **読み取り専用**: コードの変更は一切行わない
- **DB操作**: EXPLAIN や SHOW INDEX 等の読み取りコマンドのみ実行可。INSERT/UPDATE/DELETE/ALTER は禁止
- **推測と事実の区別**: DB接続なしで分析した場合、「推定」であることを明記する
- **計測ベース**: 可能な限り具体的な数値（行数、テーブルサイズ推定、クエリ実行時間）を示す
