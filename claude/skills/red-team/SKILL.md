# セキュリティ Red Team 分析

5つの攻撃者ペルソナが、それぞれ異なる動機・手法で対象を分析する。
品質検証（/review）とは相補的な関係にあり、「悪意ある利用者の視点」に特化する。

対象: **$ARGUMENTS**

## 動的コンテキスト

```
リポジトリルート: !`git rev-parse --show-toplevel 2>/dev/null || echo "(リポジトリ外)"`
現在のブランチ: !`git branch --show-current 2>/dev/null || echo "(detached)"`
プロジェクト構造: !`git ls-files --cached 2>/dev/null || echo "(git管理外)"`
```

---

## Phase 0: 入力検証・攻撃者選択

### 0-1. 引数の解析

`$ARGUMENTS` を以下のルールで解析する:

| 引数パターン | 対象 |
|-------------|------|
| ファイルパス（`.ts`, `.py`, `.go` 等） | 指定ファイルとその入力境界フロー |
| ディレクトリパス | ディレクトリ内のエントリポイントを探索 |
| 機能名（例: `認証`, `決済`, `API`） | 関連コードを Grep/Glob で特定 |
| データフロー記述（例: `ユーザー入力 → DB保存`） | 該当フローのコードを追跡 |
| (空) | エラー → 使い方案内 |

引数が空の場合、使い方を案内して終了:
```
使い方:
  /red-team src/auth/           — 認証モジュールをRed Team分析
  /red-team 決済フロー           — 決済関連コードの攻撃面を分析
  /red-team src/api/routes.ts   — APIエンドポイントの脆弱性分析
  /red-team ユーザー登録→メール送信  — データフロー単位で分析
```

### 0-2. 入力境界フローの特定

対象コードを読み取り、**外部からデータが入ってくるポイント**（入力境界）を特定する:

- HTTPリクエスト（パラメータ、ヘッダー、ボディ）
- ファイルアップロード
- WebSocket メッセージ
- 環境変数・設定ファイル
- データベースからの読み取り（他システムが書き込んだデータ）
- 外部API のレスポンス
- ユーザー生成コンテンツ（UGC）

### 0-3. 攻撃者ペルソナの選択

AskUserQuestion で分析に使用するペルソナを確認する:

```
分析対象の入力境界フローを特定しました:
{特定したフロー一覧}

以下の攻撃者ペルソナから、実行するものを選択してください（複数可、カンマ区切り）:

1. A1: 不正利用者 — STRIDE分析、認証バイパス、インジェクション（技術的脆弱性）
2. A2: 悪用者 — ビジネスロジック悪用、100倍テスト（ルールの抜け穴）
3. A3: アンチ・批判者 — UX の穴、ポリシー乖離、ダークパターン（信頼毀損）
4. A4: 荒らし — スパム、通報悪用、コミュニティ破壊（妨害行為）
5. A5: 競合 — 情報漏洩、技術偵察、APIレスポンス分析（情報搾取）
6. ALL: 全ペルソナで分析（推奨）

選択 [1-6]:
```

## Phase 1: 構造的前提の抽出

サブエージェント実行前に、プロジェクトの構造的前提を収集する。

### 1-1. アーキテクチャ情報の取得

以下を Grep/Glob/Read で調査する:
- `ARCHITECTURE.md`, `README.md`, `docs/` 内の設計ドキュメント
- 認証・認可の実装パターン（ミドルウェア、ガード）
- エラーハンドリングのパターン
- バリデーションの共通処理
- レート制限の実装
- ログ出力のパターン

### 1-2. 前提の整理

収集した情報を以下の形式で整理する（サブエージェントへの入力となる）:

```
[構造的前提]
- 認証方式: {JWT / セッション / API キー / etc.}
- 認可モデル: {RBAC / ABAC / etc.}
- バリデーション: {共通ライブラリ / 個別実装}
- レート制限: {有 (実装方式) / 無}
- エラーハンドリング: {共通ミドルウェア / 個別実装}
- ログ: {構造化ログ / 非構造化 / 不明}
```

## Phase 2: サブエージェント並列実行

Phase 0 で選択されたペルソナごとに、Agent ツールで並列実行する。

### サブエージェントへの指示テンプレート

各サブエージェントに以下の指示を渡す:

```
あなたは Red Team のセキュリティ分析者です。

## 実行指示書
まず以下のファイルを Read で読み取り、あなたのペルソナと分析手法を理解してください:
{リポジトリルート}/.config/claude/skills/red-team/red-team-agent.md

## あなたのペルソナ
{ペルソナID}: {ペルソナ名}

## 分析対象
{対象ファイル/フロー}

## 入力境界フロー
{Phase 0 で特定したフロー}

## 構造的前提
{Phase 1 で整理した前提}

## ルール
- 実行指示書の統一出力フォーマットに従って結果を返すこと
- 最大3シナリオ。深さ優先で分析すること
- コードを編集しないこと（読み取り専用）
- 推測は「可能性がある」と明示すること
```

### 並列実行の制御

- 選択されたペルソナ数に応じて Agent を並列起動する（最大5並列）
- 各エージェントは独立して動作し、互いの結果に依存しない

## Phase 3: 結果マージ・優先度付与

### 3-1. 結果の収集

全サブエージェントの結果を収集し、統合する。

### 3-2. 誤検知フィルタリング

以下の基準で誤検知を判定する:

- **フレームワークが防御済み**: 使用フレームワークの標準機能で既に対策されている場合
- **到達不能パス**: 実際にはコードパスが到達できない場合
- **前提条件が非現実的**: 攻撃の前提条件が極めて非現実的な場合

誤検知と判定したシナリオは `rejected` に分類し、理由を明記する。

### 3-3. 優先度付与

残ったシナリオを以下の2カテゴリに分類する:

| カテゴリ | 基準 | アクション |
|---------|------|----------|
| **now** | CRITICAL/HIGH かつ攻撃が容易 | 即時対応が必要 |
| **next** | MEDIUM/LOW または攻撃に高い前提が必要 | 次のスプリントで対応 |

## Phase 4: Pre-Mortem 分析

「6ヶ月後、このシステムでセキュリティインシデントが発生した。何が起きたか？」

Phase 3 の結果を踏まえ、最も起こりうるインシデントシナリオを1-2件描写する:

```markdown
### Pre-Mortem: 6ヶ月後のインシデントシナリオ

#### シナリオ: {インシデント名}
- **発生日**: 202X年Y月Z日（仮想）
- **概要**: {何が起きたか}
- **攻撃経路**: {Phase 3 のどのシナリオが現実化したか}
- **影響範囲**: {ユーザー数、データ量、ビジネス影響}
- **根本原因**: {なぜ防げなかったか}
- **教訓**: {今のうちに何をすべきだったか}
```

## 品質制御ルール

- **入力境界フロー重視**: ファイル単位ではなくデータフロー単位で分析する。
- **深さ優先**: 浅い指摘を大量に出すより、実際に悪用可能な具体的シナリオを少数精鋭で。
- **誤検知の排除**: フレームワーク標準の防御を考慮し、実際に機能する攻撃のみを報告する。
- **accepted/rejected の明確化**: なぜ有効/無効と判断したかの理由を明記する。
- **推測の明示**: 確証がない場合は「可能性がある」と明示する。
- **建設的**: 脆弱性の指摘と共に、具体的な対策を提案する。

---

## 制約事項（厳守）

- **ファイルの編集禁止**: 脆弱性の指摘のみを行い、コードを書き換えない。
- **Git操作禁止**: commit, push, reset 等の状態変更コマンドは実行しない。
- **攻撃の実行禁止**: 脆弱性の分析のみ行い、実際の攻撃（エクスプロイト実行）は行わない。
- **推測の明示**: 確証がない推測は必ず「可能性がある」等と明示する。
- **機密情報配慮**: コード中のAPIキー、パスワード等を不必要に出力しない。
- **教育的目的**: すべての指摘は防御の改善を目的とし、攻撃の詳細手順の公開を目的としない。

## 調査の進め方

- まず対象コードの入力境界を特定し、データフローを追跡する。
- サブエージェントは最大5並列で起動し、各ペルソナの分析を独立して実行する。
- Agent（Explore）は、構造把握や広範な検索が必要な場合に使用する。
- 結果のマージ時に重複シナリオを統合し、誤検知をフィルタリングする。
